AWS ECS Interview Questions & Answers
100 Questions from Beginner to Expert

🟢 BEGINNER (Questions 1–25)

1. What is Amazon ECS?
Amazon Elastic Container Service (ECS) is a fully managed container orchestration service provided by AWS. It allows you to run, stop, and manage Docker containers on a cluster of EC2 instances or using AWS Fargate (serverless). ECS handles scheduling, scaling, health monitoring, and integration with other AWS services.
2. What does ECS stand for?
ECS stands for Elastic Container Service.
3. What is a container in the context of AWS ECS?
A container is a lightweight, standalone, executable package that includes everything needed to run a piece of software — code, runtime, libraries, and settings. In ECS, containers are typically Docker containers that run based on Docker images stored in registries like Amazon ECR or Docker Hub.
4. What is a Docker image?
A Docker image is a read-only template used to create containers. It contains the application code, dependencies, environment variables, and configuration. Images are built from a Dockerfile and can be stored in container registries like Amazon ECR (Elastic Container Registry).
5. What is a Task Definition in ECS?
A Task Definition is a JSON blueprint that describes how one or more containers should run. It specifies: Docker image to use, CPU and memory requirements, networking mode, IAM roles, environment variables, port mappings, logging configuration, and volume mounts. It's similar to a Kubernetes Pod spec.
6. What is an ECS Cluster?
An ECS Cluster is a logical grouping of tasks or services. It acts as the infrastructure boundary for your containerized workloads. Clusters can use EC2 instances (where you manage the underlying infrastructure) or Fargate (serverless, where AWS manages it). A single AWS account can have multiple clusters.
7. What is an ECS Service?
An ECS Service ensures that a specified number of task instances run continuously. If a task fails or stops, the service scheduler automatically launches another instance to replace it. Services also support load balancing, rolling deployments, and auto scaling.
8. What is the difference between a Task and a Service in ECS?
A Task is a single running instance of a Task Definition — it starts, runs, and stops. It's used for one-off or batch jobs. A Service is a long-running abstraction that maintains a desired count of tasks, automatically replaces failed tasks, integrates with load balancers, and manages deployments. Use Tasks for short-lived work and Services for always-on workloads.
9. What launch types does ECS support?
ECS supports two launch types: • EC2 Launch Type: You provision and manage a cluster of EC2 instances. You have more control over the underlying infrastructure, instance types, and cost optimization. • Fargate Launch Type: Serverless. AWS manages the underlying infrastructure. You only define CPU and memory per task. You don't manage servers, scaling of hosts, or patching.
10. What is AWS Fargate?
AWS Fargate is a serverless compute engine for containers that works with both ECS and EKS. With Fargate, you don't need to provision or manage EC2 instances. You simply define your task's CPU and memory requirements, and AWS handles everything else — server provisioning, patching, scaling, and isolation using Firecracker microVMs.
11. What is the difference between EC2 launch type and Fargate launch type?
EC2 Launch Type: You manage EC2 instances (capacity, patching, scaling). More control, better for optimizing costs at scale. Supports GPU workloads. Fargate Launch Type: Serverless — no EC2 management. Pay per task (vCPU + memory seconds). Faster to get started, better security isolation. Slightly higher per-unit cost but no idle capacity waste. Choose EC2 for high-density, cost-sensitive workloads; Fargate for simplicity and isolation.
12. What is Amazon ECR?
Amazon Elastic Container Registry (ECR) is a fully managed Docker container registry provided by AWS. It allows you to store, manage, and deploy Docker images securely. ECR integrates natively with ECS and IAM for access control. It supports image scanning for vulnerabilities, lifecycle policies to auto-delete old images, and replication across regions.
13. How do you push a Docker image to Amazon ECR?
Steps to push to ECR: 1. Authenticate Docker: aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin <account_id>.dkr.ecr.us-east-1.amazonaws.com 2. Build the image: docker build -t my-app . 3. Tag it: docker tag my-app:latest <account_id>.dkr.ecr.us-east-1.amazonaws.com/my-app:latest 4. Push it: docker push <account_id>.dkr.ecr.us-east-1.amazonaws.com/my-app:latest
14. What is the ECS Agent?
The ECS Agent is an open-source software component that runs on each EC2 instance in an ECS cluster. It communicates with the ECS control plane to receive task instructions, report task status, manage container lifecycle, and publish metrics. It is pre-installed on ECS-optimized AMIs and runs as a Docker container itself.
15. What operating systems can run the ECS Agent?
The ECS Agent runs on Amazon Linux, Amazon Linux 2, and other Linux distributions (Ubuntu, RHEL, CentOS). It also supports Windows Server via a Windows-specific agent. ECS Anywhere extends this to on-premises Linux and Windows servers.
16. What is a container instance in ECS?
A container instance is an EC2 instance that is registered with an ECS cluster and is running the ECS Agent. It provides the compute resources (CPU, memory, network) on which ECS tasks are placed and run. You are responsible for managing, patching, and scaling container instances when using the EC2 launch type.
17. What is the default network mode for ECS tasks?
The default network mode is bridge mode (on EC2 launch type). In bridge mode, containers share the EC2 instance's network interface via Docker's built-in bridge networking. Port mappings are used to expose container ports to the host. For Fargate, awsvpc is the only supported (and default) network mode.
18. What are the network modes available in ECS Task Definitions?
ECS supports these network modes: • bridge: Default for EC2. Uses Docker bridge networking with port mappings. • host: Container uses the host's network namespace directly (no port mapping overhead). Only for EC2. • awsvpc: Each task gets its own Elastic Network Interface (ENI) and private IP. Required for Fargate. Recommended for EC2 as well. • none: No external network connectivity for the container.
19. What is a Task IAM Role?
A Task IAM Role is an IAM role assigned to an ECS task (not the EC2 instance). It grants containers within the task specific AWS permissions. For example, a task can be given permission to read from S3 or write to DynamoDB. This follows the least-privilege principle — each task only gets the permissions it needs.
20. What is the difference between a Task Role and a Task Execution Role?
Task Role (taskRoleArn): Permissions for the application code running inside the container — e.g., access to S3, DynamoDB, Secrets Manager. Task Execution Role (executionRoleArn): Permissions for the ECS Agent and Fargate infrastructure to set up the task — e.g., pulling images from ECR, writing logs to CloudWatch, retrieving secrets from Secrets Manager at startup.
21. How do you define CPU and memory in ECS?
CPU and memory can be defined at two levels: • Task level: Hard limit for the entire task. Required for Fargate (uses valid vCPU/memory combinations like 0.25 vCPU / 512 MB). • Container level: Soft and hard limits per container. Hard limit (memory) kills the container if exceeded. Soft limit (memoryReservation) is used for scheduling decisions. Fargate requires task-level CPU/memory. EC2 uses task-level for scheduling and container-level for enforcement.
22. What is the minimum memory you can allocate to a Fargate task?
The minimum memory for a Fargate task is 512 MB. This corresponds to the smallest valid Fargate configuration: 0.25 vCPU and 512 MB memory. Fargate CPU and memory must be chosen from predefined combinations (e.g., 0.25 vCPU supports 512 MB, 1 GB, or 2 GB).
23. What is Amazon ECS-optimized AMI?
The ECS-optimized AMI is a pre-configured EC2 Amazon Machine Image provided by AWS. It comes with the ECS Agent pre-installed, Docker Engine configured, and other dependencies set up. It's the recommended base for EC2 container instances. AWS regularly releases new versions with security patches and agent updates. You can find the latest AMI ID via SSM Parameter Store.
24. How do you view logs from an ECS container?
Logs can be viewed via: • CloudWatch Logs: Configure the awslogs log driver in the Task Definition. Logs are sent to a specified CloudWatch log group. • aws logs get-log-events CLI command. • AWS Console: Navigate to ECS → Cluster → Task → Container → Logs tab. • ECS Exec: Run interactive commands inside a running container. • Third-party tools: Datadog, Splunk, Fluentd (using log routing sidecars).
25. What is AWS CloudWatch Logs integration with ECS?
ECS integrates with CloudWatch Logs via the awslogs log driver defined in the Task Definition's logConfiguration section. You specify the log group name, region, and optional stream prefix. The ECS Agent (or Fargate) forwards stdout/stderr from containers to CloudWatch. You can then set retention policies, create metric filters, and trigger alarms based on log content.
🟡 INTERMEDIATE (Questions 26–60)

26. How does ECS Service Auto Scaling work?
ECS Service Auto Scaling uses AWS Application Auto Scaling to adjust the desired task count of an ECS service. You define scaling policies (Target Tracking, Step Scaling, or Scheduled Scaling) linked to CloudWatch metrics. When a metric threshold is crossed, the policy adjusts the task count up or down. The ECS service scheduler then launches or stops tasks accordingly.
27. What metrics can trigger ECS Auto Scaling?
Common metrics include: • ECSServiceAverageCPUUtilization: Scales based on average CPU usage across tasks. • ECSServiceAverageMemoryUtilization: Scales on memory usage. • ALBRequestCountPerTarget: Scales based on requests per task from an ALB Target Group. • Custom CloudWatch metrics: Any custom metric you publish (e.g., queue depth, active connections). • SQS ApproximateNumberOfMessagesVisible: Common for worker patterns.
28. What is the difference between Target Tracking and Step Scaling in ECS?
Target Tracking: You set a target value (e.g., keep CPU at 50%). Auto Scaling automatically adjusts task count to maintain that target. Simpler to configure and preferred for most use cases. Step Scaling: You define step adjustments — e.g., 'add 2 tasks when CPU > 70%, add 4 tasks when CPU > 90%'. Gives more granular control over how aggressively to scale. Better for workloads with sudden, unpredictable bursts.
29. How does ECS integrate with an Application Load Balancer (ALB)?
When creating an ECS Service, you can attach it to an ALB Target Group. ECS automatically registers and deregisters task IPs with the Target Group as tasks start and stop. The ALB routes incoming traffic to healthy tasks using path-based or host-based routing. With awsvpc mode, each task gets its own IP registered directly. This enables zero-downtime deployments and health-check-based traffic routing.
30. What is a Target Group in the context of ECS?
An ALB Target Group is a logical grouping of targets (in ECS's case, task IPs and ports) that receive traffic from the ALB. ECS services register their tasks as targets automatically. The ALB performs health checks on each target and only routes traffic to healthy ones. You can have multiple Target Groups for different services (path-based routing).
31. How does ECS handle service discovery?
ECS supports service discovery via AWS Cloud Map. When configured, ECS automatically registers task IPs as DNS records in a Cloud Map namespace. Other services can discover them via DNS (e.g., myservice.local). ECS also supports service connect (built-in service mesh lite) for intra-cluster communication with load balancing and retries, without needing an ALB.
32. What is AWS Cloud Map and how does it relate to ECS?
AWS Cloud Map is a managed service discovery service. It creates and manages DNS records and health checks for services. ECS integrates with Cloud Map to automatically register/deregister task IPs when tasks start/stop. This allows services to find each other using DNS names rather than hardcoded IPs, enabling dynamic service-to-service communication.
33. What is the awsvpc network mode and why is it important?
In awsvpc mode, each ECS task gets its own Elastic Network Interface (ENI) with a dedicated private IP address. This provides: • True network isolation between tasks • Security groups can be applied directly to tasks (not just the EC2 host) • Consistent behavior between EC2 and Fargate • Better compatibility with VPC flow logs for auditing It's the only mode supported by Fargate and is recommended for production EC2 workloads.
34. How does ECS assign ENIs when using awsvpc mode?
When a task with awsvpc mode is launched, ECS requests an ENI from AWS. The ENI is attached to the EC2 instance (or managed by Fargate) and the task's containers use it for all network communication. ECS interfaces with the VPC networking stack to assign a private IP, apply security groups, and configure routing. Each EC2 instance has a limit on the number of ENIs, which limits the number of awsvpc tasks per instance.
35. What is a Placement Strategy in ECS?
A Placement Strategy determines how ECS places tasks on container instances (EC2 launch type only). There are three types: • spread: Distributes tasks evenly across instances or AZs for high availability. • binpack: Packs as many tasks as possible on the fewest instances to minimize cost. • random: Places tasks randomly across available instances. You can combine multiple strategies for nuanced placement logic.
36. What are the types of ECS Task Placement Strategies?
The three placement strategy types are: • spread: Evenly distributes tasks across a specified field (availabilityZone, instanceId, or custom attribute). Best for HA. • binpack: Places tasks on the instance with the least available resource (CPU or memory). Minimizes the number of instances used. • random: Places tasks on any available instance randomly. Lowest overhead, no optimization.
37. What is a Task Placement Constraint?
Placement Constraints are rules that filter which container instances are eligible for task placement. Two types: • distinctInstance: Ensures each task is placed on a different container instance. • memberOf: Uses cluster query language expressions to match instances with specific attributes, e.g., place only on instances with attribute:ecs.instance-type =~ t3.* or in a specific AZ.
38. How do you pass environment variables to an ECS container?
Environment variables can be passed via the Task Definition in several ways: • environment: Hardcoded key-value pairs (visible in plain text in the Task Definition). • secrets: References to AWS Secrets Manager or SSM Parameter Store. ECS resolves these at task startup and injects them as env vars. Secrets are never stored in the Task Definition itself. • environmentFiles: Reference to an .env file stored in S3 (bulk import of environment variables).
39. What is AWS Secrets Manager integration with ECS?
ECS can retrieve secrets from AWS Secrets Manager and inject them as environment variables or in the secrets block of a container definition. At task startup, the ECS Agent (via the execution role) calls Secrets Manager's GetSecretValue API to fetch the secret value and provide it to the container. This avoids storing sensitive data in task definitions or application code.
40. How do you use AWS Parameter Store with ECS Task Definitions?
In the Task Definition's secrets field, specify the SSM Parameter Store parameter ARN and the environment variable name. Example: {name: 'DB_PASSWORD', valueFrom: 'arn:aws:ssm:us-east-1:123456789:parameter/my-app/db-password'}. The Task Execution Role must have ssm:GetParameters permission. ECS fetches the value at task launch and injects it as an environment variable.
41. What is a sidecar container pattern in ECS?
A sidecar is a secondary container in the same task that provides supporting functionality to the main application container. Common examples: • Log router (Fluent Bit): Collects and forwards logs. • Service mesh proxy (Envoy): Handles inter-service communication. • AWS X-Ray daemon: Collects distributed traces. • Metrics exporter: Exposes Prometheus metrics. Sidecar containers share the task's network and can share volumes with the main container.
42. How do you configure container dependencies in a Task Definition?
Use the dependsOn field in the container definition to control startup and shutdown order. Each dependency specifies: • containerName: The container this one depends on. • condition: START (other container has started), COMPLETE (exited successfully), SUCCESS (exited with code 0), or HEALTHY (passed health check). Example: A main app can depend on an init container with condition: SUCCESS before starting.
43. What is the essential flag in a container definition?
The essential flag (boolean) marks a container as critical to the task. If true and the container stops or crashes, the entire task is stopped. At least one container per task must be essential. Non-essential containers (e.g., log shippers, metric collectors) can fail without killing the task. This is important for sidecar patterns where sidecar failure shouldn't kill the application.
44. What happens when an essential container in a task fails?
When an essential container exits (regardless of exit code), ECS stops all other containers in the task and the task itself transitions to a STOPPED state. If the task belongs to a Service, the service scheduler detects the stopped task and launches a replacement task to maintain the desired count, subject to the deployment configuration and health check settings.
45. How do ECS rolling updates work?
When you update an ECS Service (e.g., new Task Definition), ECS performs a rolling update: 1. Launches new tasks with the updated definition up to maximumPercent. 2. Waits for new tasks to become healthy (pass health checks). 3. Deregisters old tasks from the load balancer (connection draining). 4. Stops old tasks. The rate is controlled by minimumHealthyPercent and maximumPercent settings.
46. What is the minimum healthy percent and maximum percent in ECS service deployment?
minimumHealthyPercent: The lower bound on the number of running tasks during deployment, as a percentage of the desired count. E.g., 50% means ECS can stop tasks until only half the desired count remains. maximumPercent: The upper bound on running tasks during deployment. E.g., 200% means ECS can launch up to double the desired count temporarily. These two settings control the pace and resource usage of rolling deployments.
47. What are ECS deployment circuit breakers?
ECS deployment circuit breakers monitor the health of a rolling deployment. If a certain threshold of tasks fail to reach a RUNNING state within a time window, the circuit breaker triggers and rolls back the deployment to the last stable task definition automatically. This prevents a bad deployment from taking down your entire service. You can configure it with optional rollback enabled in the deployment configuration.
48. How does ECS integrate with AWS CodePipeline for CI/CD?
A typical ECS CI/CD pipeline with CodePipeline: 1. CodeCommit/GitHub: Source code trigger. 2. CodeBuild: Build Docker image, push to ECR. 3. CodeDeploy or ECS Deploy Action: Update ECS Service with new Task Definition revision. CodePipeline has a native ECS deploy action that updates the service's task definition. For blue/green, CodeDeploy manages traffic shifting between old and new tasks.
49. What is the ECS Exec feature?
ECS Exec allows you to run interactive commands or open a shell inside a running container — similar to 'kubectl exec' in Kubernetes or 'docker exec' locally. It uses AWS Systems Manager Session Manager under the hood, so no SSH or open ports are needed. Commands: aws ecs execute-command --cluster my-cluster --task task-id --container app --interactive --command '/bin/bash'
50. How do you enable ECS Exec on a task?
To enable ECS Exec: 1. Task Execution Role must have SSM permissions (ssm:StartSession, ssm:TerminateSession, etc.). 2. The service or standalone task must have enableExecuteCommand: true. 3. The SSM Agent must be available (built into Fargate; must install on EC2). 4. The container must have a valid shell (e.g., /bin/bash or /bin/sh). 5. Use the AWS CLI: aws ecs execute-command --cluster X --task Y --container Z --command '/bin/sh' --interactive
51. What is a capacity provider in ECS?
A capacity provider is an ECS abstraction that links a cluster to a specific compute resource. For EC2, it links to an Auto Scaling Group and optionally enables Cluster Auto Scaling (managed scaling of EC2 instances). For Fargate, the built-in FARGATE and FARGATE_SPOT providers are available. Capacity providers enable more sophisticated compute management and cost optimization than using launch types directly.
52. What is the difference between a managed and unmanaged capacity provider?
Managed capacity provider (EC2): ECS controls the Auto Scaling Group — it scales in/out EC2 instances based on task demand using managed scaling. ECS handles the math of how many instances are needed. Unmanaged capacity provider (EC2): You link an ASG but ECS does not control scaling. You manually manage the EC2 instance count. Fargate providers are always managed (AWS controls everything).
53. What is Cluster Auto Scaling in ECS?
Cluster Auto Scaling (CAS) automatically adjusts the number of EC2 instances in a capacity provider's Auto Scaling Group based on task demand. When tasks can't be placed due to insufficient capacity, CAS scales out instances. When instances are underutilized, CAS scales them in. It uses a Capacity Provider Reservation metric to determine the scaling target.
54. What is a Capacity Provider Strategy?
A Capacity Provider Strategy defines how tasks are distributed across multiple capacity providers when running tasks or services. Each entry has: • capacityProvider: The provider name (e.g., FARGATE, FARGATE_SPOT, my-ec2-asg). • weight: Relative proportion of tasks to place on this provider. • base: Minimum number of tasks to always place on this provider. Example: base=1 on FARGATE and weight=4 on FARGATE_SPOT runs 1 guaranteed task on Fargate and the rest on Spot.
55. How do you set a default capacity provider strategy for a cluster?
When creating or updating an ECS cluster, you can set defaultCapacityProviderStrategy. This strategy is used when tasks or services are launched without specifying a launch type or capacity provider strategy explicitly. You set it via the AWS Console, CLI (aws ecs put-cluster-capacity-providers), or IaC tools like CloudFormation or Terraform.
56. What is the difference between REPLICA and DAEMON scheduling strategies in ECS?
REPLICA: ECS maintains a specified desired count of tasks across the cluster. Tasks can be placed on any available container instance. Suitable for stateless services like APIs. DAEMON: ECS runs exactly one task on every active container instance in the cluster. New instances automatically get the daemon task. Suitable for infrastructure tasks like log collectors, monitoring agents, or service mesh proxies that should run on all nodes.
57. When would you use a DAEMON service?
DAEMON services are used for infrastructure-level workloads that should run on every EC2 container instance: • Log collection agents (Fluent Bit, Fluentd, CloudWatch Agent) • Monitoring/metrics agents (Datadog agent, Prometheus node exporter) • Security/compliance tools (AWS Inspector agent, Aqua, Twistlock) • Service mesh data plane proxies • Vulnerability scanning agents Note: DAEMON is not supported on Fargate since you don't manage instances.
58. How do you configure health checks for ECS containers?
Container-level health checks are defined in the Task Definition's healthCheck field: • command: Shell command that returns 0 (healthy) or 1 (unhealthy). • interval: Time between checks (default 30s). • timeout: Max time for the command to complete (default 5s). • retries: Number of failures before marking unhealthy (default 3). • startPeriod: Grace period before health checks start (useful for slow-starting apps). ALB health checks are separate and configured on the Target Group.
59. What is the difference between ECS health checks and ALB health checks?
ECS container health checks: Run inside the container (a shell command). Used to determine if the container process is healthy. Results affect task status. ALB health checks: HTTP/HTTPS/TCP checks performed by the ALB on the task's exposed port. Used to determine if traffic should be sent to the task. Tasks failing ALB health checks are deregistered from the target group. Both can coexist. ECS health checks are for the container process; ALB health checks are for network-level availability.
60. How do volume mounts work in ECS Task Definitions?
Volumes are defined at the task level and mounted in container definitions: • Bind mounts: Map a path on the host to a container path. Ephemeral, shared between containers in the task. • Docker volumes: Managed Docker volumes. Persist for the task's lifetime. • EFS volumes: Persistent, shared storage via Amazon Elastic File System. Available on both EC2 and Fargate. Survives task restarts. • Secrets volumes: Used to inject secrets as files. You define volumes in the volumes array and reference them in mountPoints in container definitions.
🟠 ADVANCED (Questions 61–85)

61. How does ECS handle blue/green deployments using AWS CodeDeploy?
ECS + CodeDeploy blue/green works as follows: 1. A new Task Definition revision is created (green environment). 2. CodeDeploy creates a new Target Group and registers the green tasks. 3. The ALB listener rule is updated to shift traffic (all-at-once, canary, or linear). 4. After validation, old (blue) tasks are terminated. The AppSpec file defines hooks (BeforeAllowTraffic, AfterAllowTraffic) for pre/post deployment validation. If tests fail, CodeDeploy rolls back by shifting traffic back to blue.
62. What is the difference between rolling update and blue/green deployment in ECS?
Rolling Update: Gradually replaces old tasks with new ones within the same target group. Simpler, no extra infrastructure needed. Risk: both versions run simultaneously during deployment. Blue/Green (CodeDeploy): Creates a completely separate 'green' environment. Traffic switches atomically (or gradually). The 'blue' environment remains live for instant rollback. Safer for production but requires more infrastructure and ALB listener management.
63. How do you implement canary deployments with ECS?
Using CodeDeploy with ECS, select a canary traffic shifting configuration: CodeDeployDefault.ECSCanary10Percent5Minutes sends 10% of traffic to new tasks for 5 minutes, then shifts 100% if healthy. You can also implement canary deployments manually with weighted target groups on an ALB — sending a small percentage to a new service and the rest to the old one, then gradually increasing the weight.
64. How does ECS integrate with AWS App Mesh?
AWS App Mesh is a service mesh that uses Envoy proxy sidecars. ECS integrates via: 1. An Envoy sidecar container is added to each task. 2. The Envoy proxy intercepts all inbound/outbound traffic. 3. App Mesh control plane (via appmesh.local) distributes routing configuration to Envoy sidecars. 4. Traffic management (retries, timeouts, weighted routing), observability (distributed tracing, metrics), and security (mTLS) are handled at the proxy layer without app code changes.
65. What is a Virtual Node in App Mesh in the context of ECS?
A Virtual Node in App Mesh represents a logical pointer to an ECS service (or any compute backend). It defines: • The service discovery name (DNS or Cloud Map) for the service. • Health check configuration. • Backend virtual services it communicates with. Each ECS service maps to one App Mesh Virtual Node. Envoy proxies use Virtual Node configuration to know how to find and connect to backend services.
66. How would you implement a service mesh pattern with ECS and Envoy?
Steps to implement: 1. Create App Mesh mesh, Virtual Nodes, Virtual Routers, and Virtual Services. 2. Add Envoy as a sidecar container in each Task Definition. 3. Configure Envoy with the mesh and virtual node via environment variables (APPMESH_RESOURCE_ARN). 4. Set the Envoy container as a proxy configuration (proxyConfiguration in Task Definition) to intercept traffic. 5. Enable App Mesh add-on for X-Ray integration for distributed tracing. 6. Apply mTLS between virtual nodes using ACM Private CA for secure service-to-service communication.
67. What is the significance of task revision numbers in ECS?
Every time a Task Definition is modified and registered, it gets a new revision number (e.g., my-task:1, my-task:2). Revisions are immutable — you cannot modify an existing revision. ECS services point to a specific revision (or LATEST). This enables rollbacks (point the service to a previous revision), audit trails (what changed between revisions), and controlled deployments (promote specific revisions through environments).
68. How does ECS handle task networking at scale with IP exhaustion in awsvpc mode?
In awsvpc mode, each task requires an ENI and a private IP from the VPC subnet. EC2 instances have a per-instance ENI limit. At scale, you can exhaust VPC IPs or ENI slots. Solutions: • Use large VPC CIDR ranges and subnet sizes. • Use VPC CNI prefix delegation to assign /28 prefixes to ENIs (dramatically increases IPs per ENI). • Spread workloads across multiple subnets in different AZs. • Consider trunk ENI (network interface sharing feature for ECS EC2) to support more tasks per instance.
69. What is VPC CNI prefix delegation and how does it help ECS?
Prefix delegation assigns a /28 IPv4 prefix (16 IPs) or /80 IPv6 prefix to each ENI on an EC2 instance instead of individual secondary IPs. This allows many more tasks per instance without requiring additional ENIs. For ECS on EC2 with awsvpc mode, this can increase tasks-per-instance by 14x compared to individual IP assignment. It reduces ENI consumption and helps with IP address management at scale.
70. How would you architect a multi-region ECS deployment?
Key components: • Deploy identical ECS clusters in each region using IaC (Terraform modules/CDK stacks). • Use Amazon Route 53 with latency-based or geolocation routing to direct users to the nearest region. • Replicate container images to regional ECR repositories using ECR cross-region replication. • Use DynamoDB Global Tables or Aurora Global Database for multi-region data. • Implement active-active or active-passive failover depending on RPO/RTO requirements. • Use AWS Global Accelerator for anycast IPs and faster failover at the network layer.
71. How does ECS integrate with AWS Global Accelerator?
AWS Global Accelerator provides static anycast IP addresses that route traffic over the AWS backbone to the nearest healthy endpoint. For ECS, you configure Global Accelerator with ALB endpoints in multiple regions as endpoint groups. Global Accelerator performs health checks and automatically routes users to the nearest healthy ALB/ECS service. This reduces latency and provides faster failover than DNS-based routing.
72. How do you implement zero-downtime deployments in ECS?
Key practices for zero-downtime deployments: 1. Use rolling updates with minimumHealthyPercent: 100 (never reduce below desired count). 2. Configure proper ALB health checks and connection draining (deregistration delay). 3. Implement container health checks with an appropriate startPeriod. 4. Use deployment circuit breakers for auto-rollback on failures. 5. Implement graceful shutdown in your application (handle SIGTERM, drain connections). 6. Consider blue/green deployment for truly seamless traffic switching.
73. What are the trade-offs between ECS and EKS?
ECS Pros: Simpler, tighter AWS integration, less operational overhead, native IAM/ALB/CloudWatch integration, no need to manage a control plane. ECS Cons: AWS-proprietary (vendor lock-in), less feature-rich than Kubernetes, limited extensibility. EKS Pros: Industry-standard Kubernetes, portable workloads, rich ecosystem (Helm, operators, CRDs), better for complex multi-team platforms. EKS Cons: Significantly more complex, steeper learning curve, higher operational overhead, control plane costs. Choose ECS for: AWS-native teams, simpler microservices. Choose EKS for: Kubernetes expertise, portability, complex scheduling needs.
74. When would you choose ECS over EKS, and vice versa?
Choose ECS when: • Team is new to containers and wants simplicity. • You want tight AWS-native integration with minimal configuration. • You don't need Kubernetes-specific features (custom resources, operators, etc.). • Using Fargate as primary compute for full serverless containers. Choose EKS when: • You have Kubernetes expertise or are migrating from on-prem Kubernetes. • You need portability and want to avoid vendor lock-in. • You require Kubernetes ecosystem tools (Istio, Argo CD, Keda, etc.). • Complex multi-tenant platform with namespaces and RBAC needs.
75. How does ECS handle stateful workloads with EFS?
For stateful workloads requiring persistent storage: 1. Create an Amazon EFS file system in the same VPC. 2. In the Task Definition, define an EFS volume (specify file system ID, root directory, IAM authorization, and encryption in transit). 3. Mount the volume to a container path. 4. Multiple tasks (even across AZs and instances) can mount the same EFS file system simultaneously. This pattern is used for CMS files, ML model storage, shared configuration, and any workload needing persistent, shared storage.
76. How do you mount an Amazon EFS volume in an ECS Fargate task?
In the Task Definition: 1. Add to volumes: {name: 'efs-vol', efsVolumeConfiguration: {fileSystemId: 'fs-12345', transitEncryption: 'ENABLED', authorizationConfig: {accessPointId: 'fsap-xxx', iam: 'ENABLED'}}} 2. In container definition mountPoints: [{sourceVolume: 'efs-vol', containerPath: '/data', readOnly: false}] 3. The Task Execution Role needs elasticfilesystem:ClientMount permission. 4. Ensure EFS security group allows NFS (port 2049) from the task's security group.
77. How would you debug a task that keeps stopping unexpectedly?
Debugging approach: 1. Check the stopped task's stoppedReason and containerExitCode in the ECS console or CLI. 2. Check CloudWatch Logs for application errors. 3. Common causes: Out of memory (OOM kill — increase memory), application crash (check logs), health check failures (tune startPeriod, check the health check command), missing IAM permissions (check execution role), container image pull failure (check ECR permissions), EFS mount failure. 4. Use ECS Exec to get into a running container before it crashes. 5. Check CloudTrail for API permission errors. 6. Review CPU throttling metrics in CloudWatch.
78. How do you handle secret rotation in ECS without restarting tasks?
ECS injects secrets at task startup — it does not automatically refresh them. For secret rotation without task restarts: • Implement secret caching in your application with TTL-based refresh using AWS SDK (SecretsManager.GetSecretValue). • Use the AWS Secrets Manager caching library which auto-refreshes on rotation. • For database credentials, use RDS Proxy which handles credential rotation transparently. • If you must use env vars, design your service for graceful rolling restarts triggered by a deployment pipeline when secrets rotate.
79. How does ECS Fargate Spot work and what are its trade-offs?
Fargate Spot uses spare AWS compute capacity at up to 70% discount compared to regular Fargate. AWS can reclaim Spot capacity with a 2-minute warning (SIGTERM sent to containers). Trade-offs: Pros: Significant cost savings, good for batch jobs, fault-tolerant stateless services, and dev/test workloads. Cons: Tasks can be interrupted at any time. Not suitable for stateful workloads, long-running single-instance jobs, or latency-sensitive production traffic without proper graceful shutdown handling.
80. How do you handle Fargate Spot interruptions gracefully?
Best practices for Fargate Spot interruptions: 1. Handle SIGTERM: Application should catch SIGTERM and drain in-flight requests, then exit cleanly within 2 minutes. 2. Use multiple tasks: Never run a single Spot task for critical work — use replicas so remaining tasks handle traffic. 3. Mix Spot and regular Fargate: Use capacity provider strategy with base=1 on FARGATE and remaining tasks on FARGATE_SPOT. 4. Design stateless services: No local state to lose on interruption. 5. Use SQS for work queues: Unprocessed messages return to the queue after visibility timeout.
81. What is the ECS Task Metadata Endpoint and how is it used?
The ECS Task Metadata Endpoint is an HTTP endpoint available to containers at runtime (via the ECS_CONTAINER_METADATA_URI environment variable). It provides: • Task-level metadata: task ARN, cluster, task definition, resource limits. • Container-level metadata: container name, image, Docker ID. • Task and container-level statistics: CPU, memory, network metrics (compatible with Docker stats API). Applications use it for self-identification, telemetry, and dynamic configuration based on task context.
82. How do you implement distributed tracing with ECS using AWS X-Ray?
Steps: 1. Add X-Ray daemon as a sidecar container in the Task Definition. 2. Instrument your application with the AWS X-Ray SDK (automatic or manual instrumentation). 3. The SDK sends trace segments to the X-Ray daemon on UDP port 2000. 4. The daemon forwards traces to the X-Ray service. 5. The Task Role needs xray:PutTraceSegments and xray:PutTelemetryRecords permissions. 6. View traces in AWS X-Ray console to see service maps, latency distributions, and error analysis.
83. How do you configure an X-Ray daemon as a sidecar in ECS?
In the Task Definition, add a container: • image: amazon/aws-xray-daemon • cpu: 32, memoryReservation: 256 (low resource allocation) • portMappings: [{containerPort: 2000, protocol: 'udp'}] • environment: [{name: 'AWS_REGION', value: 'us-east-1'}] The main application container sends traces to localhost:2000 (in awsvpc mode since containers share the task network namespace). Set AWS_XRAY_DAEMON_ADDRESS=localhost:2000 in the app container.
84. What are the limits of ECS and how do you work around them?
Key limits: • Tasks per cluster: 5,000 (request increase via Service Quotas). • Services per cluster: 5,000 (same). • Container definitions per task: 10. • ENIs per EC2 instance: Varies by instance type (use prefix delegation to increase effective IP count). • Task definition size: 64 KB. • Fargate max CPU: 16 vCPU, max memory: 120 GB. Work-arounds: Use multiple clusters for large deployments, split large task definitions, use EKS for extreme scale, request quota increases proactively.
85. How do you implement cross-account ECR image access in ECS?
To pull ECR images from another AWS account: 1. In the source account, add a registry policy on the ECR repository granting the target account's ECS Task Execution Role ecr:GetDownloadUrlForLayer, ecr:BatchGetImage, ecr:BatchCheckLayerAvailability. 2. In the Task Definition, reference the full image URI including the source account ID. 3. The Task Execution Role in the target account must have the cross-account pull permission (granted via the ECR repository policy, not an IAM policy on the role). 4. Ensure VPC connectivity if using PrivateLink for ECR.

86. How would you design an ECS architecture for a high-traffic, low-latency microservices platform?
Architecture considerations: • Compute: Fargate for operational simplicity or EC2 with CAS for cost optimization at scale. • Networking: awsvpc mode for task-level security groups, PrivateLink for AWS service access. • Load balancing: ALB with path-based routing, connection multiplexing, and HTTP/2 support. 87. Explain how ECS Anywhere works and its use cases.
ECS Anywhere extends AWS ECS to run on any on-premises or non-AWS infrastructure (bare metal, VMs, edge devices). How it works: 1. Install the ECS Agent and SSM Agent on your server. 2. Register the server as an EXTERNAL container instance in an ECS cluster. 3. Deploy ECS tasks to those servers just like EC2 container instances. Use cases: • Hybrid cloud: Run containers on-premises with centralized management from AWS. • Data sovereignty: Keep processing on-premises for compliance. • Edge computing: Run containers on edge servers close to data sources. • Migration: Gradually move workloads to AWS without full re-architecture.
88. How do you manage ECS workloads on on-premises hardware using ECS Anywhere?
Setup process: 1. Create an ECS cluster with EXTERNAL instances. 2. Generate an activation via SSM (aws ssm create-activation --iam-role ECSAnywhereIAMRoleInstances). 3. On the on-premises server, run the provided curl/install script with the activation ID and code. 4. The server registers as an SSM Managed Instance and ECS External container instance. 5. Deploy tasks using the EXTERNAL launch type. Management: All task scheduling, deployment, and observability goes through the ECS control plane in AWS. Network connectivity between on-premises and AWS needed for SSM/ECS control plane communication.
89. What security hardening steps would you take for an ECS production environment?
Comprehensive security hardening: • IAM: Least-privilege task roles and execution roles per service. No wildcards. • Networking: awsvpc mode, security groups per service, private subnets, no public IPs on tasks. • Secrets: Secrets Manager/SSM Parameter Store (never env var hardcoding). Enable rotation. • Images: Scan ECR images (ECR Enhanced Scanning with Inspector), use minimal base images (distroless), pin image digests not tags. • Runtime: Read-only root filesystems, drop Linux capabilities, no privileged containers. • Logging: Enable CloudTrail, VPC Flow Logs, CloudWatch Logs for all containers. • Network: Use PrivateLink for AWS service access (no internet egress needed). • Encryption: Enforce TLS for all service-to-service communication via App Mesh mTLS.
90. How do you implement network segmentation and least-privilege networking in ECS?
Strategies: • Security groups per service: In awsvpc mode, assign unique security groups to each ECS service. Services only open ports to specific other services' security groups, not CIDR ranges. • Subnet segmentation: Place different service tiers in different subnets (frontend, backend, data tier). • Network ACLs: Add coarse-grained subnet-level firewall rules as a second layer. • No public IPs: All ECS tasks in private subnets with NAT Gateway for outbound-only internet access. • PrivateLink: Access AWS services (ECR, Secrets Manager, S3) via VPC endpoints without internet traffic. • App Mesh mTLS: Mutual TLS ensures only authorized services can communicate, even within the VPC.
91. How does AWS PrivateLink work with ECS and ECR?
PrivateLink creates VPC Interface Endpoints that allow traffic to AWS services to stay within the AWS network without traversing the public internet. For ECS: • com.amazonaws.region.ecr.dkr: For Docker pull operations. • com.amazonaws.region.ecr.api: For ECR API calls. • com.amazonaws.region.ecs-agent, ecs-telemetry, ecs: For ECS Agent communication. • com.amazonaws.region.secretsmanager: For secret retrieval. • com.amazonaws.region.logs: For CloudWatch Logs. With these endpoints, ECS tasks in private subnets don't need a NAT Gateway to operate, improving security and reducing data transfer costs.
92. How do you optimize ECS costs for a large-scale production workload?
Cost optimization strategies: • Fargate Spot: Use for interruptible workloads (batch, background jobs) with a capacity provider strategy mix. • EC2 Spot Instances: Use Spot Fleet with managed capacity providers for EC2 launch type. • Savings Plans: Compute Savings Plans cover Fargate usage (not EC2 Spot). • Right-sizing: Analyze ECS Container Insights metrics to find over-provisioned tasks. Reduce CPU/memory reservations. • Binpack placement: Use binpack strategy to maximize EC2 instance utilization. • Scale to zero: For non-production environments, schedule tasks to stop at night (EventBridge Scheduler → Lambda → ECS update-service desired=0). • Graviton: Use ARM64 (Graviton) instances or Fargate ARM for up to 20% cost savings.
93. How would you implement autoscaling based on SQS queue depth for ECS?
Architecture: 1. ECS worker service consumes messages from an SQS queue. 2. Create a CloudWatch custom metric tracking ApproximateNumberOfMessagesVisible / number of running tasks (backlog per task). 3. Configure Target Tracking scaling policy targeting this metric (e.g., keep backlog at 100 messages per task). 4. Alternatively, use KEDA (Kubernetes-style) via Lambda: EventBridge rule triggers Lambda to check SQS depth and update ECS desired count. 5. For Fargate: same approach works since scaling is at task level. 6. Set scale-in cooldown generously to avoid thrashing on bursty queues.
94. What is the role of AWS EventBridge in ECS-based architectures?
EventBridge integrates with ECS in several ways: • ECS State Change Events: ECS publishes task and service state change events to EventBridge (task started, stopped, service stable). Use rules to trigger Lambda for alerting, remediation, or automation. • Scheduled Tasks: EventBridge Scheduler can trigger ECS RunTask API on a schedule (cron jobs, ETL jobs). • Pipe integration: EventBridge Pipes can consume from SQS/Kinesis and trigger ECS tasks with the event payload. • Cross-account/cross-region eventing: Forward ECS events to other accounts for centralized monitoring.
95. How do you use ECS with AWS Step Functions for batch processing pipelines?
Step Functions integrates with ECS via the EcsRunTask state. The pipeline: 1. Step Functions defines workflow states (data extraction, transformation, validation, load). 2. Each state triggers an ECS RunTask (Fargate) passing parameters via input/output. 3. Step Functions waits for task completion using .sync:2 integration (polling) or heartbeat tokens. 4. On success/failure, Step Functions transitions to the next state or error handler. 5. Parallel states can run multiple ECS tasks concurrently. 6. This pattern replaces complex Lambda orchestration for compute-intensive batch jobs that exceed Lambda's 15-minute limit.
96. How do you prevent container breakout attacks in ECS?
Container security hardening to prevent breakout: • Never run containers as root: Set user in Dockerfile or task definition (user: '1000'). • Read-only root filesystem: Set readonlyRootFilesystem: true in container definition. • Drop Linux capabilities: Use linuxParameters.capabilities.drop: ['ALL'] and add back only what's needed. • No privileged mode: Never set privileged: true unless absolutely necessary. • No host networking/pid sharing: Don't use host network mode or pidMode: host. • Use Fargate: Provides VM-level isolation (Firecracker) so container breakout can't affect other customers. • Runtime security: Tools like Falco, Aqua, or Sysdig detect anomalous container behavior at runtime.
97. Explain how Fargate's micro-VM isolation (Firecracker) enhances security.
AWS Fargate uses Firecracker, an open-source VMM (Virtual Machine Monitor) developed by AWS. Each Fargate task runs in its own Firecracker microVM with a minimal kernel. Key security properties: • Strong isolation: Unlike containers sharing a kernel, each Fargate task has its own kernel. A kernel exploit in one task cannot affect another. • Minimal attack surface: Firecracker has ~50K lines of code vs millions for traditional hypervisors (KVM, Xen). • Bare-metal performance: Near-native performance with VM-level security. • No shared kernel vulnerabilities: Container breakout exploits (runc vulnerabilities, etc.) don't apply across Fargate tasks. This makes Fargate suitable for multi-tenant workloads with strong security requirements.
98. How would you implement a custom ECS scheduler?
ECS supports custom schedulers via the StartTask API (place tasks on specific instances). Implementation: 1. Build a service (Lambda, EC2, Fargate) that:   a. Calls DescribeContainerInstances to get available instances and their resources.   b. Applies custom placement logic (GPU availability, custom attributes, cost-based placement).   c. Calls StartTask API with the chosen instance ID. 2. Register custom attributes on instances (aws ecs put-attributes) for the scheduler to query. 3. Use EventBridge to trigger the scheduler on demand or periodically. Use cases: GPU-aware placement, rack-aware placement for latency, license-aware scheduling, compliance-based instance selection.

100. If an ECS service is stuck in a deployment and not reaching steady state, walk through your complete troubleshooting process.
Complete troubleshooting process: Step 1 — Check service events: In the ECS console, go to the service's Events tab. Look for 'service failed to launch a task', health check failures, or port conflicts. Step 2 — Inspect stopped tasks: Find recently stopped tasks. Check stoppedReason and each container's exitCode and reason. Exit code 1 = app error. Exit code 137 = OOM kill. Exit code 139 = segfault. Step 3 — Check CloudWatch Logs: Look at the application logs for the failed tasks. Look for startup errors, missing configs, or connection failures. Step 4 — Check health checks: ALB health checks — is the target group showing tasks as unhealthy? Is the health check path correct? Is the startPeriod in the container health check long enough? Step 5 — Check IAM permissions: CloudTrail — are there AccessDenied events? Does the execution role have ECR pull permissions? Does the task role have required service permissions? Step 6 — Check networking: Can the task reach its dependencies (DB, cache, APIs)? Check security group rules. Are VPC endpoints configured if internet access is blocked? Step 7 — Check resource availability: Is the cluster full? For EC2, are container instances showing insufficient CPU/memory? For Fargate, is there a service quota issue? Step 8 — Check image: Does the ECR image tag exist? Is it the right architecture (arm64 vs x86_64)? Step 9 — Use ECS Exec: If a task is briefly running, use ECS Exec to inspect the container state interactively. Step 10 — Rollback: If the deployment is failing, manually update the service to the previous task definition revision or let the circuit breaker handle automatic rollback.
